# PoisonedFinance — Deployment (Fly.io) + CI (GitHub Actions) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Containerise the API with a multi-stage Docker build, deploy it to Fly.io (free tier) with migrations running automatically on each release, and wire up GitHub Actions CI that runs the full test suite (unit + BDD) on every push/PR plus a deploy workflow that ships to Fly.io on merge to main.

**Architecture:** The API is packaged as a Docker image built from `api/Dockerfile` using a two-stage Node 20 build (install + compile → lean runtime). `fly.toml` at the repo root drives Fly.io config; a `[deploy] release_command` runs `node dist/db/migrate-cli.js` before traffic is shifted to the new release, satisfying contracts §11 (migrations run on release). GitHub Actions uses two workflow files: `ci.yml` (test on every push/PR, including non-blocking `@wip` BDD per contracts §14) and `deploy.yml` (Fly.io deploy on push to main or manual trigger). Secrets are stored in Fly.io (via `fly secrets set`) and in GitHub (via repo Secrets for `FLY_API_TOKEN`); the `.env.example` documents every required variable.

**Tech Stack:** Docker (multi-stage, Node 20 alpine), Fly.io free tier (`flyctl`), GitHub Actions, `superfly/flyctl-actions`, Jest + ts-jest (unit), jest-cucumber (BDD features per §14), `js-yaml` (CI YAML validation), actionlint (optional local lint)

### Task 1: API Dockerfile

**Files:**
- Create: `api/Dockerfile`

The Dockerfile uses two stages. Stage `builder` installs all dependencies (including devDependencies for `tsc`) and compiles TypeScript. The `npm run build` script from the API scaffold plan runs `tsc && cp -r src/db/migrations dist/db/migrations`, so migration SQL files land in `dist/db/migrations` where `migrate.ts` and `migrate-cli.ts` both expect them (via `path.join(__dirname, 'migrations')`). Stage `runtime` starts from the same base but copies only `dist/` and `package.json`, then runs `npm install --omit=dev` for a lean production image.

- [ ] **Step 1: Create `api/Dockerfile`**

```dockerfile
# ── Stage 1: builder ──────────────────────────────────────────────────────────
FROM node:20-alpine AS builder

WORKDIR /app

# Copy manifests first for layer-cache efficiency
COPY package.json ./

# Install ALL deps (including devDependencies for tsc)
RUN npm install

# Copy source and compile
COPY tsconfig.json ./
COPY src/ ./src/

# Produces dist/ with JS + copies src/db/migrations → dist/db/migrations
RUN npm run build

# ── Stage 2: runtime ──────────────────────────────────────────────────────────
FROM node:20-alpine AS runtime

WORKDIR /app

# Copy manifests for production install
COPY package.json ./

# Install production deps only
RUN npm install --omit=dev

# Copy compiled output (includes dist/db/migrations)
COPY --from=builder /app/dist ./dist

# Fly.io injects PORT; default to 3000 for local docker run
ENV PORT=3000
EXPOSE 3000

CMD ["node", "dist/index.js"]
```

// without actually connecting to a database or exiting the test process.

jest.mock('@/db/migrate', () => ({ runMigrations: mockRunMigrations }));

const mockExit = jest.spyOn(process, 'exit').mockImplementation((_code?: string | number | null | undefined) => {

  return undefined as never;

// Suppress console output during tests for clean output
const mockConsoleLog = jest.spyOn(console, 'log').mockImplementation(() => undefined);
const mockConsoleError = jest.spyOn(console, 'error').mockImplementation(() => undefined);

describe('migrate-cli', () => {
  beforeEach(() => {
    jest.resetModules();
    mockRunMigrations.mockReset();
    mockExit.mockClear();
    mockConsoleLog.mockClear();
    mockConsoleError.mockClear();

    mockExit.mockRestore();
    mockConsoleLog.mockRestore();
    mockConsoleError.mockRestore();

### Task 6: CI workflow

**Files:**
- Create: `.github/workflows/ci.yml`

The CI workflow runs on every push and pull_request. It has two jobs:

1. `test` — sets up Node 20, runs `npm ci && npm test` for both `api/` and `mobile/`. Uses a matrix so both run in parallel and failures are reported independently.
2. `bdd` — runs jest-cucumber feature specs (contracts §14). `@wip` scenarios are run in a separate step that is `continue-on-error: true` so they are non-blocking until the corresponding plan lands.

- [ ] **Step 1: Create `.github/workflows/ci.yml`**

```yaml
# .github/workflows/ci.yml
# Runs on every push and pull request.
# Jobs: test (api + mobile unit tests), bdd (jest-cucumber feature specs).
# BDD @wip scenarios are non-blocking (continue-on-error: true) per contracts §14.

name: CI

on:
  push:
    branches: ["**"]
  pull_request:
    branches: ["**"]

jobs:
  # ── Unit tests (API + Mobile) ───────────────────────────────────────────────
  test:
    name: Unit tests — ${{ matrix.workspace }}
    runs-on: ubuntu-latest

    strategy:
      fail-fast: false
      matrix:
        workspace: [api, mobile]

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Set up Node.js 20
        uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"
          cache-dependency-path: ${{ matrix.workspace }}/package.json

      - name: Install dependencies
        run: cd ${{ matrix.workspace }} && npm install

      - name: Run tests
        run: cd ${{ matrix.workspace }} && npm test

  # ── BDD feature specs (jest-cucumber, contracts §14) ───────────────────────
  bdd:
    name: BDD feature specs
    runs-on: ubuntu-latest
    # BDD job does not block merge — it is informational until all scenarios land.
    # Remove this line once all @wip tags are cleared.

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Set up Node.js 20
        uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"
          cache-dependency-path: api/package.json

      - name: Install API dependencies
        run: cd api && npm install

      - name: Run BDD specs (non-@wip) — blocking
        # Runs only the implemented scenarios; failure blocks CI.
        run: cd api && npm test -- --testPathPattern="__tests__/features" --testNamePattern="^(?!.*@wip)"

      - name: Run BDD specs (@wip) — non-blocking
        # Runs @wip-tagged scenarios; failure is reported but does not block merge
        # until the corresponding implementation plan lands (contracts §14).
        continue-on-error: true
        run: cd api && npm test -- --testPathPattern="__tests__/features" --testNamePattern="@wip"
```

### Task 7: Deploy workflow

**Files:**
- Create: `.github/workflows/deploy.yml`

The deploy workflow runs on push to `main` (the default branch) and supports manual dispatch via `workflow_dispatch`. It uses the official `superfly/flyctl-actions/setup-flyctl` action to install flyctl, then runs `flyctl deploy --remote-only` which sends the Dockerfile to Fly.io's remote builders (no Docker daemon needed on the runner). The `FLY_API_TOKEN` secret must be added to the GitHub repo's secrets (Settings → Secrets and variables → Actions).

**Branch note:** This plan is being written on a feature branch. The `push: branches: [main]` trigger means the deploy workflow fires only after this branch is merged to `main`. The workflow_dispatch trigger allows manual deploys at any time once flyctl is configured.

- [ ] **Step 1: Create `.github/workflows/deploy.yml`**

```yaml
# .github/workflows/deploy.yml
# Deploys to Fly.io on push to main (after PR merge) or manual dispatch.
#
# Prerequisites:
#   1. fly secrets set ... (Task 5) — all runtime secrets in Fly.io.
#   2. FLY_API_TOKEN set in GitHub repo Secrets
#      (Settings → Secrets and variables → Actions → New repository secret).
#      Generate token: fly tokens create deploy -x 999999h --app poisonedfinance-api
#   3. Fly.io app created: fly apps create poisonedfinance-api --org personal
#   4. Fly.io Postgres attached: fly postgres attach <pg-app-name> --app poisonedfinance-api
#
# The release_command in fly.toml runs migrations before traffic shifts.
# A migration failure aborts the deploy and keeps the previous release live.

name: Deploy to Fly.io

on:
  push:
    branches:
      - main
  workflow_dispatch:
    inputs:
      reason:
        description: "Reason for manual deploy"
        required: false
        default: "manual trigger"

jobs:
  deploy:
    name: Deploy — poisonedfinance-api
    runs-on: ubuntu-latest
    environment: production

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Set up flyctl
        uses: superfly/flyctl-actions/setup-flyctl@master

      - name: Deploy to Fly.io (remote build)
        # --remote-only: Fly.io builds the Docker image on its own builders.
        # No local Docker daemon required on the GitHub Actions runner.
        # The Dockerfile is at api/Dockerfile; fly.toml specifies dockerfile = "api/Dockerfile".
        run: flyctl deploy --remote-only --app poisonedfinance-api
        env:
          FLY_API_TOKEN: ${{ secrets.FLY_API_TOKEN }}
```

### Automated deploy

Push to (or merge a PR into) `main`. The `.github/workflows/deploy.yml` workflow runs `flyctl deploy --remote-only` automatically.

Requires `FLY_API_TOKEN` set in GitHub repo Secrets:
```bash
fly tokens create deploy -x 999999h --app poisonedfinance-api
# Copy output → GitHub Settings → Secrets → Actions → FLY_API_TOKEN
```
